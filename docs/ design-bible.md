# Portfolio Design Bible

> "I don't build because something doesn't exist.
> I build because I'm curious why it works."

---

# Philosophy

This portfolio is not meant to be a resume.

It is meant to be a window into how I think.

Every page should answer one question:

> "What kind of engineer is she?"

Rather than

> "What technologies does she know?"

The website should communicate curiosity, thoughtfulness, storytelling and craftsmanship.

---

# Core Design Principles

- Curiosity over credentials.
- Questions over answers.
- Story over résumé.
- Calm over flashy.
- Soft over futuristic.
- Minimal over crowded.
- Every animation must have a purpose.
- Every project explains **why**, not just **what**.
- Every page should feel intentional.
- Leave enough whitespace for thoughts to breathe.

---

# Typography Rules

- Large headings.
- Comfortable reading width.
- Sentence case only.
- Avoid ALL CAPS.
- Use italics to create emotion.
- Let whitespace create hierarchy instead of bold text.
- Typography should carry the design more than colors.

---

# Signature Rule

Every page begins with a thought.

Not a heading.

Not a title.

A thought.

The content simply continues that thought.

Examples

Projects

"Some ideas became products.
Others became lessons.
Every one of them started with a question."

Photography

"Sometimes I notice the little contradictions people walk past."

Writing

"I write to understand.
Publishing is just a side effect."

Experience

"Every place taught me a different way to ask better questions."

About

"I don't think curiosity is something you have.
I think it's something you practice."

# About

Purpose

Tell my story.

Not my career timeline.

- Why I build
- What fascinates me
- What kind of problems I enjoy solving
- My interests outside engineering
- Curiosity

---

# Experience

Purpose

A concise summary.

Not LinkedIn.

Each experience should answer

"What did this teach me?"

Instead of

"What was my designation?"

Include

- Company
- Duration
- Few meaningful points
- Technologies only where relevant

---

# Projects

Purpose

The heart of the website.

Every project should feel like a case study.

Each project includes

- Opening thought
- Problem
- Why I built it
- Background
- Architecture
- Pipeline
- Challenges
- Decisions made
- Learnings
- Future improvements
- Gallery
- Videos
- GitHub
- Technical write-up

Projects should tell stories.

Not list features.

# Non-AI Chatbot

Purpose

Demonstrate engineering.

Not AI.

The chatbot should be built using

- Rules
- Intent matching
- State machine
- Context
- Carefully designed responses

Visitors should discover that

> "This isn't AI."

And still enjoy talking to it.
